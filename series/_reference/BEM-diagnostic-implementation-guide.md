# BEM診断シート 自動化フロー実装ガイド

連載「気づきから組み立てる」のリードマグネット運用に向けた、Phase 0 立ち上げのための完全実装ガイド。

---

## 1. 全体アーキテクチャ

```
[読者] → /series/diagnostic ページにアクセス
   ↓
[Tally フォーム] 8問の質問 + メアド入力
   ↓ (Webhook)
[Make.com シナリオ]
   ├─ 1. データ受信・正規化
   ├─ 2. スコア計算 (E1, E2, E3, I1)
   ├─ 3. タイプ判定 (A〜E)
   ├─ 4. Google Drive から該当PDFを取得
   ├─ 5. Gmail で結果メール送信(PDF添付)
   └─ 6. Notion / メルマガサービスにリード追加
       ↓
[読者] 受信箱で結果PDFを受け取る
   ↓
[フォローアップ] 数日後に関連記事のリマインダー (オプション)
```

すべて **Make.com の無料プラン (1,000 ops/月)** で十分回ります。Phase 0 は月20〜50件の診断を想定しているため、十分余裕。

## 2. ツール構成

| 役割 | ツール | プラン | 月額目安 |
|---|---|---|---|
| フォーム | **Tally** | Free | ¥0 |
| 自動化 | **Make.com** | Free (1,000 ops/月) | ¥0 |

**初期コスト: 0円**で運用開始できます。

### ツール選定の理由

- **Tally**: 無料で無制限フォーム作成、Webhook対応、フォームの埋め込みも可能。Typeformより無料制約が緩い
- **Make.com**: Yasuhitoさんの既存スタック。条件分岐(Router)による5タイプ判定が直感的に書ける
- **Google Drive**: Make.comとの連携が安定。PDFを差し替えても自動で最新版が送られる
- **Gmail**: 独立コンサルとして、自分のドメインメール(または Gmail)から送るのが一番信頼感がある

### 3.2 フォーム構成 (全11ブロック)

#### Block 0: イントロ

#### Block 1〜8: 8つの質問

各質問は **Multiple Choice (Single Select)** で作成。質問文と選択肢は LP の設計に従う(下記コピペ用):

---

**Q1. 期待水準の明示度 (E1)**

```
質問:
自社の現場で、社員一人ひとりに対して「何を、どの水準で、いつまでに」期待しているかは、どの程度明示されていますか?

選択肢:
A. 全員に明確に言語化・共有されている
B. 一部のチームでは明確だが、全社で揃ってはいない
C. 上司の暗黙知に依存していて、本人は自分の期待水準を完全には把握していない
D. ほとんど明示されていない、現場任せになっている
```

---

**Q2. フィードバックの頻度と質 (E1)**

```
質問:
上司から部下への業務フィードバックは、どのくらいの頻度・粒度で行われていますか?

選択肢:
A. 日々または週次で、具体的な行動レベルでフィードバックされている
B. 月次・四半期で行われ、事実ベースで議論される
C. 半期/年次評価のタイミングのみ、抽象的なコメントが多い
D. ほとんど発生していない、または雑談化している
```

---

**Q3. マネジャーの育成時間 (E2)**

```
質問:
あなたの組織で、マネジャー(管理職)が部下の育成・指導に充てる時間は、業務時間のうちどの程度ですか?

選択肢:
A. 業務時間の20%以上
B. 10-20%程度
C. 5-10%程度(プレイングマネジャー化が顕著)
D. ほぼゼロ。マネジャー自身がプレイヤー業務に追われている
```

---

**Q4. ツール・プロセスの妨げ度 (E2)**

```
質問:
現場社員が「業務を進めようとして、システムや手続きの遅さで足を引っ張られる」と感じる頻度は?

選択肢:
A. 稀。プロセスはスムーズに動いている
B. 月に数回程度
C. 週に何度も発生する
D. 日常的に妨げになっている
```

---

**Q5. 評価と「望ましい行動」の整合 (E3)**

```
質問:
自社で「望ましい」とされる行動(リーダーシップ、育成、挑戦、対話など)は、評価・処遇に明確に反映されていますか?

選択肢:
A. 行動が明示され、評価・昇進・報酬に直結している
B. 行動は言語化されているが、評価への反映は限定的
C. 「あるべき姿」としては掲げているが、実際には数字成果のみで評価されている
D. 望ましい行動と評価がほぼ無関係。または逆向きにインセンティブが働いている
```

---

**Q6. 育成 vs 数字 (E3)**

```
質問:
あなたの組織で、優秀なマネジャーとして評価されているのは「部下を育てる人」と「自分で数字を作る人」、どちらの傾向が強いですか?

選択肢:
A. 育てる人が明確に評価されている
B. どちらも評価されるが、最終的に育成が重視される
C. 短期数字を出す人が優先的に評価されている
D. 数字一辺倒で、育成行動はほぼ評価されない
```

---

**Q7. モチベーション低下の帰因 (E3)**

```
質問:
社員のモチベーション低下に直面したとき、その原因は「個人の意欲の問題」と「組織構造の問題」のどちらが大きいと感じますか?

選択肢:
A. 主に個人の意欲の問題。マインドセット研修や動機づけが効きそう
B. 半々ぐらい
C. 主に組織構造の問題。動機づけ研修だけでは解決しない
D. 完全に組織構造の問題。個人責任に帰着させると逆効果
```

---

**Q8. スキル不足の実感 (I1) ※逆配点**

```
質問:
現在パフォーマンスに課題のある社員・組織について、本当に「知識・スキル不足」が原因だと感じますか?

選択肢:
A. 明確にスキル不足。研修やOJTで埋められそう
B. スキル不足の側面はあるが、それだけでは説明できない要因がある
C. スキル自体は十分。むしろ環境・制度の問題が大きい
D. スキル不足は表層で、構造的な問題のほうが本質と感じる
```

## 5. Make.com シナリオ構築

### 5.1 シナリオ全体構成

```
[1] Webhooks: Custom webhook (受信)
    ↓
[2] Tools: Set Multiple Variables (回答→スコア変換)
    ↓
[3] Tools: Set Multiple Variables (セル別合計+正規化)
    ↓
[4] Router: 5分岐 (Type A〜E判定)
    ↓
各分岐:
[5a-e] Google Drive: Get a File (該当PDF取得)
    ↓
[6a-e] Email or Gmail: Send Email (PDF添付)
    ↓
[7] Notion: Add Database Item (リード保管・全分岐共通)
```

### 5.2 Module 1: Webhook 受信

1. Make.com で新規シナリオ
2. Trigger に「Webhooks > Custom webhook」を追加
3. 「Add hook」で新しい Webhook を作成し、URL をコピー
4. このURLを Tally の Webhook 設定に貼り付け
5. Tally でテスト送信して、Make.com が受信できるか確認

受信される JSON 構造:

```json
{
  "data": {
    "fields": [
      {"label": "Q1...", "value": "A"},
      {"label": "Q2...", "value": "B"},
      ...
      {"label": "Email", "value": "user@example.com"},
      {"label": "Name", "value": "山田太郎"},
      ...
    ]
  }
}
```

### 5.3 Module 2: スコア変換

「Tools > Set Multiple Variables」を追加。各変数を以下のように設定:

| 変数名 | 値 (式) |
|---|---|
| `q1_score` | `if(q1 = "A"; 1; if(q1 = "B"; 2; if(q1 = "C"; 3; 4)))` |
| `q2_score` | 同上 |
| `q3_score` | 同上 |
| `q4_score` | 同上 |
| `q5_score` | 同上 |
| `q6_score` | 同上 |
| `q7_score` | 同上 |
| `q8_score` | `if(q8 = "A"; 4; if(q8 = "B"; 3; if(q8 = "C"; 2; 1)))` ← **逆配点** |

※実際には Tally の field IDを使うので、`q1` の部分は適切な変数参照に置き換える。

### 5.4 Module 3: セル別合計と正規化

| 変数名 | 値 |
|---|---|
| `e1_score` | `q1_score + q2_score` |
| `e2_score` | `q3_score + q4_score` |
| `e3_score` | `q5_score + q6_score + q7_score` |
| `i1_score` | `q8_score` |
| `e1_pct` | `e1_score / 8` |
| `e2_pct` | `e2_score / 8` |
| `e3_pct` | `e3_score / 12` |
| `i1_pct` | `i1_score / 4` |

さらに **タイプ判定** の式:

```
type =
  if(i1_pct >= 0.75 AND max(e1_pct, e2_pct, e3_pct) < 0.5;
    "E";
  if((e1_pct >= 0.6 ? 1 : 0) + (e2_pct >= 0.6 ? 1 : 0) + (e3_pct >= 0.6 ? 1 : 0) >= 3;
    "D";
  if(e3_pct >= e1_pct AND e3_pct >= e2_pct;
    "C";
  if(e1_pct >= e2_pct;
    "A";
    "B"
  ))))
```

(Make.com の式言語に合わせて調整。複雑な分岐は Router で実装するほうが見やすい)

### 5.5 Module 4: Router (5分岐)

「Router」を追加し、5つのフィルタを設定:

| 分岐 | フィルタ条件 |
|---|---|
| Type A | `type = "A"` |
| Type B | `type = "B"` |
| Type C | `type = "C"` |
| Type D | `type = "D"` |
| Type E | `type = "E"` |

### 5.6 Module 5 (各分岐): Google Drive で PDF 取得

各分岐に「Google Drive > Download a File」を追加。`File ID` に対応するPDFのIDを設定。

| 分岐 | File ID |
|---|---|
| Type A | (Type A の Drive File ID) |
| Type B | (Type B の Drive File ID) |
| ... | ... |

### 5.7 Module 6 (各分岐): メール送信

「Gmail > Send an Email」(または「Email > Send an Email」)を追加。

設定:
- **To**: `{{webhook.email}}`
- **Subject**: 各タイプ別(下記テンプレート参照)
- **Content type**: HTML
- **Body**: 各タイプ別(下記テンプレート参照)
- **Attachments**: 前モジュールでダウンロードしたPDF

### 5.8 Module 7: Notion へリード保管

すべての分岐の後に合流して「Notion > Create Database Item」を追加。

Notion DB のスキーマ例:

| プロパティ | タイプ | 値 |
|---|---|---|
| 受信日時 | Date | `now()` |
| 名前 | Title | `{{webhook.name}}` |
| メアド | Email | `{{webhook.email}}` |
| 業種 | Select | `{{webhook.industry}}` |
| 役職 | Select | `{{webhook.role}}` |
| 診断結果Type | Select | `{{type}}` |
| E1スコア | Number | `{{e1_score}}` |
| E2スコア | Number | `{{e2_score}}` |
| E3スコア | Number | `{{e3_score}}` |
| I1スコア | Number | `{{i1_score}}` |
| ステータス | Select | `新規` |
| メモ | Text | (空欄、後から追記) |

### Type別の3行サマリー

**Type A (E1)**:
> 1on1が雑談化したり、評価制度が形骸化したり、目標管理が形式に流れたりする組織で頻発する型です。これは研修によるスキル付与では解決しません。期待水準とフィードバック構造の再設計が必要です。

**Type B (E2)**:
> マネジャーが慢性的に時間不足で、ツールやプロセスが業務を妨げている組織で頻発する型です。これは「タイムマネジメント研修」では解決しません。業務量・工数・プロセス設計そのものの再構築が必要です。

**Type C (E3)**:
> リーダー研修を打ってもリーダーが育たない、バリューが掲示物で終わる、育成行動が報われない、といった現象が起こる型です。これはリーダー研修やバリュー策定では解決しません。評価制度・処遇・承認構造の再設計が必要です。

**Type D (複合型)**:
> 改革を打っても局所最適化に終わったり、全方位的な機能不全に陥っている組織で見られる型です。複合型の場合、E3 → E1 → E2 の順での介入が効果的です。30分相談で介入順序の優先順位を一緒に整理することを推奨します。

**Type E (スキル特化型)**:
> 構造はおおむね機能していて、純粋にスキル不足が中心の課題と判定された稀なケースです。HPI観点で「真にスキル不足か」の再検証をしてみると、隠れたE1〜E3課題が見つかることもあります。

Make.com の Scheduler または ConvertKit のシーケンス機能で実装。

### 8.3 結果ページへの誘導 (動的)

メール送信に加えて、フォーム完了画面で結果を即時表示する。Tally の Conditional Logic + Embed コードで実装可能。

### 8.4 LP のサンクスページ最適化

メール送信を待つ間、LP 内に「待っている間に読める記事」を案内するセクションを追加。離脱を防ぐ。

### 8.5 教材販売との接続 (Phase 1)

教材ローンチ後、診断結果に応じて該当教材を案内するメールを追加。
- Type A → 1on1設計診断キット
- Type C → BEM分析Excelテンプレ
- など
